# Figma MCP

The Figma MCP server is configured project-wide in [`.mcp.json`](../../.mcp.json) at the repo root. Tools are exposed under `mcp__figma__*`.

## First-time setup

The MCP server loads automatically when the repo is opened, but each developer does two one-time things:

1. **Approve the project MCP** — on first open, Claude Code prompts you to allow the project's `.mcp.json`. Approve it.
2. **Authenticate** — call any figma tool (or run `/mcp` and select figma) and complete the Figma OAuth flow in the browser. The token caches in `~/.claude/`, which the devcontainer symlinks to `${workspaceFolder}/.claude-user/`, so auth survives container rebuilds.

If a tool call returns an auth error later, re-run `/mcp` to re-authenticate.

The plugin entry in `.claude/settings.json` (`figma@claude-plugins-official`) is kept separately because it ships additional skills and slash commands (`/figma-use`, `/figma-generate-design`, `/figma-create-new-file`) that the bare MCP server does not include. The `.mcp.json` entry is what makes the MCP tools actually register on devcontainer start.

# Figma -> Code

**Scope:** these rules apply only to the explorers-based UI stack -- `libs/explorers/`, `libs/agora/`, `libs/model-ad/`, `libs/qtl/`, and `apps/{agora,model-ad,qtl}/`. They do not apply to bixarena or any other scope (different styling, components, and conventions). If you're working outside the listed scopes, ignore this file.

When implementing a Figma design in this repo, translate Figma tokens to code using the rules below. The goal is design intent, not pixel-perfect fidelity -- prefer existing tokens, components, and icons where they're close enough, even when a few values differ. Designers do not populate Figma's Code Syntax field, so Claude does the reconciliation.

## Close enough principle

Pixel-perfect is not the goal. Skip overrides where the existing default is within a few pixels (font size, spacing, border-radius), within one shade (color), or within one weight step. The threshold for adding an override is "a designer would notice without comparing side-by-side." If yes, override; if no, leave it.

Err toward reuse. Flag to the user if unsure.

## Components

Before writing markup or styles, look for an existing component that covers the Figma element. Sources, in order:

- Scope-specific UI libs (`libs/<product>/ui/`, `libs/<product>/util/`)
- `libs/explorers/ui/` -- presentational components (header, footer, hero, search input, download utils)
- `libs/explorers/util/` -- utilities including `LoadingIconComponent` (loading spinner) and `SvgIconComponent`
- `libs/explorers/shared/` -- shared page components
- PrimeNG (the repo's component library) -- check existing usages of `<p-...>` in `libs/explorers/ui/` and `libs/explorers/comparison-tool/` to see what components are in use and which props/variants this repo relies on

If Figma shows a loading state, use `explorers-loading-icon` or PrimeNG's `loading` prop on `<p-button>` -- do not write custom `@keyframes` from scratch.

Do not introduce new UI dependencies to match a Figma design -- stick to what's already in `package.json`. For icons, see the Icons section (FontAwesome, PrimeIcons, and custom SVGs are all already available). For components, PrimeNG is the standard. If the design needs something the existing stack can't reasonably express, stop and ask the user.

### Lean on the PrimeNG preset

PrimeNG components are themed by the preset at `libs/explorers/themes/src/lib/basePreset.ts`. Accept the preset's defaults; override only when meaningfully different from Figma (close-enough principle applies).

Prefer component props and variants over CSS overrides:

- `rounded="true"` instead of `border-radius: 50px`
- `size="small"` / `size="large"` instead of pixel-level height overrides
- `variant="text"` instead of background/color overrides on `.p-button-text`
- `loading` prop instead of custom spinners

When you must override, scope tightly with a CSS class -- don't dump rules on `.p-button`, `.p-select`, `.p-popover-content` that reshape the component globally.

## Colors

`libs/explorers/styles/src/lib/_variables.scss` defines the shared token set. Read it directly to find current values -- do not assume hex values from this file, they go stale.

It contains:

- `$main-colors` -- gray scale (`gray-100`-`gray-1000`), brand palette (`primary`, `secondary`, `tertiary`, `action-primary`), status colors (`success`, `warning`, `danger`)
- `$extra-colors` -- semantic aliases (`text`, `text-secondary`, `heading`, `link`, `border`, `separator`, `shadow`, `tooltip`)

CSS custom properties are generated by `libs/explorers/styles/src/lib/_root.scss` (iterating both `$main-colors` and `$extra-colors`) as `--color-{key}`. Each product also has its own `_root.scss` (e.g. `libs/qtl/styles/src/lib/_root.scss`) that re-emits `--color-{key}` for its `$main-colors` overrides only -- so brand colors get product-specific values, while semantic aliases (`--color-text-secondary`, `--color-link`, etc.) come from explorers. Check the active per-product `_variables.scss` first when resolving a hex.

When mapping a Figma color variable to code:

1. Match the Figma hex against `_variables.scss` (including per-product overrides).
2. Prefer the **semantic** alias (`var(--color-text-secondary)`) over the raw scale entry (`var(--color-gray-800)`) when both resolve to the same hex. Semantic names survive palette changes.
3. Fall back to `var(--color-gray-*)` if no semantic alias matches.
4. Inline hex with a `// TODO: add token` comment only as a last resort, and flag it to the user.

### Known drift

Figma `Gray 200` is `#F1F2F4`; code `--color-gray-200` is `#F1F3F5`. Treat as the same token.

## Typography

Do **not** use the centralized typography scale in new code -- neither the SCSS maps (`$font-sizes`, `$headings` from `_variables.scss`) nor the CSS custom properties derived from them (`var(--font-size-md)`, `var(--line-height-md)`, `var(--font-size-h1)`, etc.). Typography is being defined per stylesheet instead.

When implementing typography from Figma, set `font-size`, `line-height`, and `font-weight` directly in the component's `.scss` file with values that match Figma (subject to the close-enough principle for line-height variations).

Use the font family already established by the active product's global styles -- check `:root` or `body` in the relevant `_root.scss` or `styles.scss` if unsure. Do not introduce a new font family without flagging to the user.

## Icons

When the Figma MCP returns an image URL (look for `const imgX = "..."` or `<img src={...}>` in generated code; the `data-name` attribute often hints at intent, e.g. `keyboard_arrow_up` -> a chevron):

1. Call `get_screenshot` to see what the asset actually looks like.
2. If it's an icon, substitute an existing one. This repo uses three icon sources -- match whatever the surrounding code/file already uses to keep consistency, then fall back to whichever fits:
   - **FontAwesome** -- `<fa-icon [icon]="faName"></fa-icon>` with `import { faName } from '@fortawesome/free-solid-svg-icons'`. The most widely used inline icon library in this repo.
   - **PrimeIcons** -- `<i class="pi pi-{name}"></i>` or `icon="pi pi-{name}"` on PrimeNG components. Use especially for icons on PrimeNG components (where PrimeIcons ships natively).
   - **Custom SVGs** via `SvgIconComponent` from `libs/explorers/util/src/lib/svg-icon/`. These live in `*-assets/icons/` directories (`explorers-assets/icons/`, `qtl-assets/icons/`, etc.). The preloaded set is in `SvgIconService.commonSvgPaths` (`libs/explorers/services/src/lib/svg-icon.service.ts`). Use for product-specific or non-standard icons.
3. If no existing icon matches (or the asset isn't an icon at all), stop and ask the user to provide the file directly from the design source. Do not fetch or commit the Figma asset URL -- those expire after ~7 days, and getting the asset from the design source ensures the committed file matches the canonical version.

## Effects

Effect variables (e.g. `Popover Drop Shadow 1`) have no code tokens. Inline the `box-shadow` matching the Figma definition. If the same shadow appears in 2+ places, flag the gap to the user.
